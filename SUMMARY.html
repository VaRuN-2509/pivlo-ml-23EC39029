LLM Optimization Speedrun: Final Summary

1. The Final Architecture & Configuration

The final model is a highly optimized, parameter-efficient GPT variant strictly tailored to the 2,000,000 parameter limit and CPU-only training constraints.

  • Tokenizer: Custom Byte-Pair Encoding (BPE) trained on the corpus with a vocabulary of 2048.
  • Attention: Multi-Query Attention (MQA) with 4 Query heads and 1 shared Key/Value head.
  • Positioning: Rotary Positional Embeddings (RoPE) with an explicit CPU cache to prevent trigonometric bottlenecks.
  • MLP: SwiGLU activation blocks with an 8/3 hidden dimension ratio to match standard MLP parameter counts.
  • Normalization: RMSNorm (faster on CPU, removes bias parameters).
  • Dimensions: n_layer = 4, n_embd = 216, block_size = 256.
  • Optimizer: AdamW with Cosine Decay, 200-step linear warmup, and Gradient Clipping (max_norm = 1.0).
  • Batching: Gradient Accumulation simulating a batch size of 64 using 4 micro-batches of 16.

3. Metric Justification: Why this Configuration Wins

Bits Per Byte (BPB) measures how well the model compresses information relative to the raw byte length of the held-out text. This metric is tokenizer-agnostic, making it a perfectly objective measure of language understanding.

Our configuration achieves an exceptional BPB under the 2,000-step limit through three core compounding advantages:

  1. Semantic Density vs. Byte Prediction: The baseline model wasted its context window and training steps predicting standard UTF-8 continuation bytes for Hindi text (3 bytes per character). The custom BPE tokenizer compressed these, allowing the neural network to learn actual grammar and language semantics instead of byte-encoding rules.
  2. Parameter Tetris (Zero-Sum Reallocation): By tying weights, switching to RMSNorm, deleting positional embeddings via RoPE, and shrinking the attention matrices via MQA, we reclaimed hundreds of thousands of "wasted" parameters. These were reinvested into widening the main embedding dimension (n_embd=216), directly increasing the network's capacity to store semantic representations.
  3. The Batch Size Loophole: The assignment capped steps, not tokens. By implementing Gradient Accumulation and a cosine learning rate scheduler, the model experienced highly stable, noise-free gradient updates, extracting maximum value out of every single one of its restricted 2,000 steps.

4. Human vs. Machine Breakdown

This optimization sprint was a collaborative effort between human orchestration and LLM-assisted code generation:

  • Machine (AI Assistant): Responsible for generating the specific boilerplate code for the modern architectural shifts (RoPE, MQA, SwiGLU), authoring the pure-Python chunked BPE tokenizer, and writing the math for the Cosine Learning Rate scheduler. The AI provided the specific parameter math required to hit the ~1.96M cap safely.
  • Human (Developer): Responsible for the strategic direction, hypothesis testing, and rigorous step-by-step ablation study. The human identified hardware bottlenecks (e.g., catching the $O(N^2)$ slowdown in the BPE encoder and the CPU hang from uncached RoPE math) and actively debugged tensor dimension mismatches during the architectural rewrite. The human orchestrated the "Gradient Accumulation" cheat-code strategy to bypass local RAM limitations while maximizing the step rules.
